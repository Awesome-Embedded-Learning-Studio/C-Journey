# 指针、内存布局与位运算：看清变量住在哪、怎么操作比特

## 引言

如果说学 C 有什么东西是"入门到放弃"的重灾区，那一定是指针。很多人背了半天"指针是地址"，但一遇到 `int* p` 还是发懵。说实话，指针之所以难，不是因为它复杂，而是因为它**太底层**——你得先在脑子里建起一张"程序在内存里长什么样"的地图，指针才有地方落脚。所以这一章我们不急着啃 `*` 和 `&`，而是先把内存这块地盘搞清楚：全局变量住哪、局部变量住哪、`malloc` 出来的又住哪。地图有了，指针就是地图上的坐标，自然就懂了。

配套示例在 [examples/stage1-armc-basics](../../examples/stage1-armc-basics/)，我们用它来印证每一条结论，而且地址都是真跑出来的。

## 核心概念：一个程序在内存里分了几个区

一个 C 程序跑起来之后，操作系统给它的那块虚拟内存并不是一坨混沌，而是分了几个各司其职的区域。我们把最要紧的几个画出来：

```text
高地址 ┌─────────────────┐
       │  栈 stack        │ ← 局部变量、函数参数、返回地址（向下生长 ↓）
       │  ...             │
       │                  │
       │  （中间空闲）     │
       │                  │
       │  堆 heap         │ ← malloc/calloc 来这里（向上生长 ↑）
       ├─────────────────┤
       │  .bss            │ ← 未初始化的全局/静态变量（自动清零）
       │  .data           │ ← 已初始化的全局/静态变量
       ├─────────────────┤
低地址 │  .text           │ ← 你的代码（机器指令）
       └─────────────────┘
```

光看图没感觉，我们写个小程序把不同变量的真实地址打出来看。下面这段分别取了全局变量、静态变量、栈上局部变量、堆上 `malloc` 出来的变量的地址：

```c
int global_var = 42;
int main(void)
{
    int stack_var = 7;
    static int static_var = 9;
    int* heap_var = (int*)malloc(sizeof(int));
    printf("global  : %p\n", (void*)&global_var);
    printf("static  : %p\n", (void*)&static_var);
    printf("stack   : %p\n", (void*)&stack_var);
    printf("heap    : %p\n", (void*)heap_var);
    free(heap_var);
    return 0;
}
```

真跑一遍，地址长这样（每次运行会变，这是 ASLR 地址随机化干的，但相对关系不变）：

```text
global  (全局/静态区) : 0x61577d497030
static  (全局/静态区) : 0x61577d497034
stack   (栈)          : 0x7fff4f644cac
heap    (堆)          : 0x6157b9564010
```

你看，`global_var` 和 `static_var` 的地址紧挨着（只差 4 字节，因为都是 `int`），它俩住在同一个区；栈变量 `stack_var` 跑到了 `0x7fff…` 这个高端地址；而 `malloc` 出来的堆变量在低端。**这就是"变量住在哪"的直观答案**——所谓指针，存的就是这些地址里的某一个。

> 顺带一提，栈是**向下生长**的（新调用的函数，局部变量地址越来越小），堆是**向上生长**的，两者从两头往中间挤。这就是为什么"栈溢出"和"堆溢出"是两个方向的事。

## 动态内存：malloc / realloc / free

栈上的变量有个特点——函数一返回就没了。当你需要一块"活得比函数久"、或者"大小运行时才知道"的内存时，就得去堆上申请，那就是 `malloc` 一族的活儿。我们看 [Exp2/heap.c](../../examples/stage1-armc-basics/Exp2/heap.c)，它把 `malloc` / `memset` / `memcpy` / `realloc` / `free` 一条龙走了一遍：

```c
p = (char*)malloc(100);          // 申请 100 字节
memset(p, 0, 100);               // 清零
memcpy(p, "hello", 5);           // 拷进去
p = (char*)realloc(p, 200);      // 扩容到 200
free(p);                         // 释放
```

真跑一遍，重点看 `realloc` 前后地址的变化：

```text
p = 0x5e08e76ac010        ← malloc 初始地址
hello                     ← memcpy 之后的内容
0x5e08e76ad090            ← realloc 之后，地址变了！
hello                     ← 内容还在
```

发现没有，`realloc` 之后地址从 `…ac010` 变成了 `…ad090`，**不是原地扩容，而是给你换了一块新地方**。这是因为原地扩容不一定行得通——如果原地址后面紧跟着的内存已经被占了，操作系统只能在堆里另找一块够大的，把老数据搬过去，再返回新地址。这也是为什么 `realloc` 的返回值必须接住，不能假定老指针还有效。

> **踩坑预警**：新手特别爱写 `p = realloc(p, newsize)`，看着没毛病，其实埋了雷。万一 `realloc` 失败返回 `NULL`，你直接赋给 `p`，原来的 `p` 就再也找不回来了——内存泄漏。正确姿势是用一个临时变量接，成功了再覆盖：
> ```c
> char* tmp = realloc(p, newsize);
> if (tmp) p = tmp;       // 失败就保留原 p，不丢数据
> ```

## 位运算：直接摸比特

搞嵌入式、搞协议解析、搞寄存器，位运算是家常便饭。[Exp3](../../examples/stage1-armc-basics/Exp3/) 给了两个特别典型的位操作宏，一个置位、一个清位：

```c
// 置位：把 x 的第 bit 位置 1
#define SET_BIT(x, bit)   (x = ((x) | (1 << bit)))
// 清位：把 x 的第 bit 位清 0
#define CLEAR_BIT(x, bit) (x = ((x) & (~(1 << (bit)))))
```

这两个宏看着吓人，拆开其实就三招：`<<` 把 1 移到对应位置、`|` 置位、`& ~` 清位。我们拿 `SET_BIT` 跑一遍，从一个 `a = 1`（二进制 `…0001`）开始，循环置位：

```text
1           ← 0b0001
3           ← 0b0011
7           ← 0b0111
15          ← 0b1111
31          ← 0b1 1111
…
```

每置一位，值就变成下一个 `2ⁿ-1`，因为低位被一个接一个填满成 1。这就是"置位"的直观效果。

你可能会问，宏里那一堆括号是干嘛的，写 `(1 << bit)` 不就够了，为啥要套成 `((x) | (1 << bit))`？这是 C 宏的生存法则——**运算符优先级会咬人**。举个例子，很多人随手写 `if (x & 1 == 0)`，以为意思是"x 的最低位是 0"，但其实 `==` 的优先级比 `&` 高，编译器看到的是 `x & (1 == 0)`，也就是 `x & 0`，永远是 0。宏里把每个参数、每个子表达式都拿括号包死，就是为了防止这种"展开后优先级错乱"的事故。

## 常见坑（真正的坑在后面）

> **坑 1：用了没声明的变量。** [Exp1/var.c](../../examples/stage1-armc-basics/Exp1/var.c) 里有个典型翻车——用了一个根本没声明的 `k`，编译器直接怼回来：
> ```text
> error: 'k' undeclared (first use in this function)
>    printf("k = %d\n", k);
> ```
> C 不像某些动态语言能"用了再说"，**变量必须先声明再使用**，而且作用域只在声明它的那对 `{}` 里有效。看到 `undeclared` 别去别处找，就是这个名字没在当前作用域里声明过。

> **坑 2：野指针 / 未初始化的指针。** `int* p;` 之后直接 `*p = 10;`，`p` 指向哪谁也不知道，写进去就是定时炸弹。指针声明了要么赋个有效地址，要么先 `= NULL`，至少崩也好崩在可预测的地方。

> **坑 3：`free` 之后忘了置空。** `free(p)` 释放了内存，但 `p` 这个变量本身的值没变，还指着那块已经还回去的内存。这时你要是不小心又 `*p` 或再 `free(p)`，就是经典的 use-after-free 和 double-free。养成习惯：`free(p); p = NULL;`。

> **坑 4：位运算和比较的优先级。** 上面讲过了，`x & mask == 0` 不是你想的那个意思，永远记得加括号 `(x & mask) == 0`。

## 小结

- [ ] 程序在内存里分区：栈（局部）、堆（malloc）、全局/静态（.data/.bss）、代码（.text）
- [ ] 指针存的就是这些区里某个变量的地址
- [ ] `realloc` 不一定原地扩容，可能搬家，返回值要用临时变量接
- [ ] 位操作三招：`<<` 移位、`|` 置位、`& ~` 清位；宏里括号不能省
- [ ] 变量先声明后用，指针要么指向有效地址要么 NULL，`free` 后立刻置空
