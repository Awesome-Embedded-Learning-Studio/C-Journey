# 从零写一个操作系统：通电到内核的接力赛

## 引言

如果你问我学计算机这一路上最"上头"的事是什么，我会说是一次性把一个操作系统从零写出来、让它在自己眼前跑起来。这不是夸张——当你第一次看到自己写的汇编字符"MBR"出现在模拟器屏幕上，那种"我真正让一块硬件听我话了"的快感，是写多少业务代码都换不来的。所以这一章不是 ROADMAP 八阶段里的必修内容，它是一个**进阶挑战**，适合在啃完前面阶段、特别是第六章裸机 C 之后，用来把所有底层知识串起来熔炼一遍。配套代码就在 [os-from-scratch](../../projects/os-from-scratch/)，从第 1 课到第 20 课逐步演进，`final/` 是一个相当完整的阶段性版本。

> 老规矩先交底：OS 代码得用 `nasm` 汇编、`dd` 写进软盘/硬盘镜像，然后在 Bochs 里跑，所以这儿没法给你贴运行截图。但下面每一行汇编和 C 都是真实的，你照着把镜像做出来、丢进 Bochs 就能看到效果。

## 核心概念：从按电源到执行你的代码

写 OS 的第一道关卡，是搞明白"按下电源到 CPU 执行我写的第一行代码"中间发生了什么。这件事在桌面编程里被操作系统和 BIOS 完全藏起来了，但在写 OS 的时候，你得亲手走完这条接力链：

```text
通电 → BIOS 自检 → BIOS 读硬盘第一个扇区(MBR)到 0x7c00 并跳过去
      → MBR(你写的) 把 loader 从硬盘读进内存并跳过去
      → loader(你写的) 进入保护模式、读 kernel 进内存并跳过去
      → kernel(你写的) 的 main 开始跑，OS 正式上线
```

关键在于：**BIOS 只帮你做第一棒**——它把硬盘的第一个扇区（512 字节，叫 MBR 主引导记录）读到内存的 `0x7c00` 位置然后跳过去，剩下的全是你自己的事。所以你的第一段代码，就是塞在这 512 字节里的 MBR。我们看 [final/boot/mbr.S](../../projects/os-from-scratch/final/boot/mbr.S) 的开头：

```asm
SECTION MBR vstart=0x7c00        ; 这段代码被加载到 0x7c00 运行
    mov ax, 0xb800
    mov gs, ax                    ; gs 指向 0xb800——VGA 文本模式显存
    ...
    mov byte [gs:0x00], 'M'       ; 往显存写字符
    mov byte [gs:0x01], 0xA4      ; 0xA4 是属性字节：红底白字
```

看到 `0xb800` 这个魔法数字了吗？在 VGA 文本模式下，**内存的 `0xb800` 段就是屏幕本身**——你往这里写一个字符字节，屏幕上就立刻显示出来，没有任何 API、没有任何驱动，内存即显示。这就是裸机编程的极致：没有抽象，你直接摸硬件。MBR 里写 `M`、`B`、`R` 三个字符，就是为了让"MBR 跑起来了"这件事肉眼可见。

然后 MBR 干完正事，最后两行特别重要：

```asm
    times 510-($-$$) db 0     ; 用 0 填充到第 510 字节
    db 0x55, 0xaa             ; 最后两字节必须是 0x55 0xaa
```

`0x55, 0xaa` 是引导扇区的**魔数**——BIOS 就是靠检查这两个字节来判断"这个扇区是不是可引导的"。少了它，BIOS 根本不跳过来执行你的 MBR。这种"约定好的魔法数字"在底层到处都是，是硬件和软件之间的暗号。

MBR 最后把自己从硬盘读出来的 loader 跳过去执行（`jmp LOADER_BASE_ADDR + 0x300`），接力棒就交给了 loader。loader 的活更重——它要带着 CPU 从 16 位实模式切到 32 位保护模式、加载 GDT、把内核读进内存，最后跳进内核。每一棒都在为下一棒铺路。

## 内核起来之后：一个 OS 都有哪些零件

接力赛跑到 kernel 的 `main`，OS 才算真正上线。我们看 [final/kernel/main.c](../../projects/os-from-scratch/final/kernel/main.c) 的开头，能一眼看出这个内核都攒了哪些子系统：

```c
int main(void) {
    put_str("KERNEL INITALIZE START\n");
    init_all();          // 一口气把所有子系统初始化好
    ...
    console_put_str("[CharlieChen114514@localhost /]$ ");  // shell 提示符
    thread_exit(running_thread(), true);
}
```

光是 `main.c` 顶上那一排 `#include`，就是一个操作系统的零件清单：

| 子系统 | 大致职责 |
|---|---|
| `interrupt` | 中断管理，搭起 IDT（中断描述符表） |
| `memory` | 物理内存 + 虚拟内存管理，分页 |
| `thread` / `process` | 内核线程与进程调度 |
| `console` / `print` | 控制台输出（还是那个 0xb800 显存的高级封装） |
| `syscall` | 系统调用——用户程序请求内核服务的入口 |
| `fs` / `dir` / `ide` | 文件系统、目录、磁盘驱动 |
| `shell` | 一个能敲命令的 shell |

更狠的是，`main.c` 里还跑了一个 `init` 进程：它先 `fork` 出一个子进程跑 `my_shell()`，父进程则在一个死循环里 `wait` 回收僵尸子进程——这不就是 Linux 的 1 号进程干的事吗？能走到这一步，说明这个 OS 已经有了进程、文件系统、shell 和系统调用，麻雀虽小五脏俱全。

## 几个绕不开的硬骨头

这些每个拎出来都能写一本书，这里只给你指个方向，知道它们是干嘛的、为什么躲不开：

- **保护模式与 GDT**：实模式只能用 1MB 内存、还没有内存保护。loader 必须切到保护模式，而切换的前提是搭一张"全局描述符表"（GDT），告诉 CPU 各段在哪、权限如何。
- **中断与 IDT**：内核得能响应时钟、键盘、系统调用这些中断。CPU 靠"中断描述符表"（IDT）找到对应处理函数，你得把这张表搭好。
- **分页**：给每个进程独立的、虚拟的内存视图，靠页表映射实现。这是"每个进程都以为自己独占内存"的底层魔术。
- **内核线程与调度**：多个执行流怎么轮转？靠时钟中断触发调度器，保存/恢复上下文（看 `thread/switch.S`）。
- **系统调用**：用户程序不能直接碰硬件，得通过 `int` 或 `syscall` 指令"陷进"内核，由内核代劳。

## 怎么读这份代码

这份代码是按课次（`1/` 到 `20/`）一步步长出来的，所以最舒服的读法是**跟着课次顺序走**，而不是一上来就扎进 `final/`。每个课次通常只加一个新特性，跑通了再进下一课，你能清楚地看到 OS 是怎么从"屏幕上显示一个字符"长成"能跑 shell 的系统"的。想看完整全貌就直接读 `final/`，从 `boot/mbr.S` 顺着调用链往下，`mbr → loader → kernel/main → init_all → 各子系统`。每个目录基本都有 `Makefile`，`make` 完用 `dd` 把产物写进镜像、Bochs 里跑调试。

## 常见坑（真正的坑在后面）

> **坑 1：`0x55, 0xaa` 魔数漏了。** BIOS 不认你的 MBR，直接报 "No bootable device"，你盯着黑屏怀疑人生。

> **坑 2：实模式切保护模式的顺序。** GDT 没加载好就 `cr0` 置位开保护模式，CPU 立刻给你一个三重错误重启。

> **坑 3：裸机没有调试器帮你。** 桌面上 `gdb` 随便用，写 OS 时你的代码在 Bochs 里跑，得用 Bochs 自带的调试器（`b` 下断点、`info reg` 看寄存器），习惯不一样。

> **坑 4：汇编和 C 的调用约定。** 内核里大量是汇编（`switch.S`、`kernel.S`）和 C 混着写，参数怎么传、栈怎么对齐，搞错了就是莫名其妙的崩溃。

## 小结

- [ ] 引导链：BIOS → MBR(0x7c00) → loader → kernel main，每棒都是你写的
- [ ] MBR 的三要素：`vstart=0x7c00`、`0xb800` 显存直写、`0x55,0xaa` 魔数
- [ ] 一个 OS 的零件：中断、内存、线程、文件系统、系统调用、shell
- [ ] 读这份代码按课次顺序最顺，从 `final/` 的 `mbr.S` 顺调用链往下
- [ ] 调试靠 Bochs 自带调试器，不是桌面 gdb
