# 裸机 C 入门：在 8051 上写一个会跑的时钟

## 引言

前面几章我们的 C 程序都跑在有操作系统的桌面上，`printf` 往屏幕吐字、`malloc` 拿内存、程序结束了 OS 帮你收拾烂摊子——一切都那么理所当然。但当你拿到一块 8051 单片机，这些"理所当然"全没了：没有屏幕（要自己驱动 LCD），没有 OS 帮你调度（你得自己写主循环），连 `main` 返回了都不知道去哪。这就是**裸机编程**——你的程序就是操作系统，直接站在硬件上。这一章我们拆 [embedded-mcu/Timer](https://github.com/Awesome-Embedded-Learning-Studio/C-Journey/tree/main/projects/embedded-mcu/Timer/) 这个真实的 LCD 时钟项目，看裸机 C 和桌面 C 到底有哪几个根本性的不同。

> 先交个底：8051 的代码得用 Keil C51 工具链编译、烧到真板子上才跑得起来，所以我没法像前几章那样在这儿给你贴一段运行输出。但下面引用的每一行代码都是真的，你拿真硬件一验证就明白了。

## 核心概念一：超级循环

桌面程序的 `main` 干完活就 `return` 了，可裸机程序的 `main` **不能返回**——它返回了 CPU 就没指令可执行，整块板子直接"死"在那。所以裸机程序的标准长这样，一个永不退出的主循环，业界戏称"超级循环"（super loop）：

```c
void main()
{
    Timer0_Init();           // 初始化硬件
    LCD_Init();
    LCD_ShowString(1,1,"Timer HERE");
    while (1) {              // 永远转下去
        keyNum = Key();      // 轮询按键
        switch (keyNum) {    // 根据按键更新状态、刷新显示
            ...
        }
    }
}
```

你看 [Timer/main.c](https://github.com/Awesome-Embedded-Learning-Studio/C-Journey/blob/main/projects/embedded-mcu/Timer/main.c) 就是这个套路：先初始化（定时器、LCD），然后 `while(1)` 里不停地轮询按键、刷新时间显示。`main` 是 `void` 的，没有 `return`——这就是裸机程序的生命周期，启动之后就在那个循环里转到断电。

## 核心概念二：操作硬件就是读写寄存器

在桌面上你想点亮一个 LED，调用个库函数就行；在单片机上，你得往一个叫**寄存器**的内存地址里写值。寄存器本质上是内存映射的硬件接口——你往这个地址写 `1`，对应的引脚就通电；写 `0`，就断电。8051 把这些寄存器名都定义在 `<REGX51.H>` 里，`TMOD`、`P2`、`EA`、`TH0` 这些看着像变量的东西，其实都是固定地址的寄存器。

配定时器的时候，你直接操作这些寄存器。看 [Timer0.c](https://github.com/Awesome-Embedded-Learning-Studio/C-Journey/blob/main/projects/embedded-mcu/Timer/Timer0.c) 的初始化：

```c
void Timer0_Init(void)
{
    TMOD &= 0xF0;   // 清掉低 4 位（定时器0 的模式位）
    TMOD |= 0x01;   // 把低 4 位设成 0001（模式1：16 位定时器）
    TL0 = 0x18;     // 定时器低 8 位初值
    TH0 = 0xFC;     // 定时器高 8 位初值
    TF0 = 0;        // 清溢出标志
    TR0 = 1;        // 启动定时器0
    ET0 = 1;        // 允许定时器0 中断
    EA = 1;         // 开总中断
}
```

这里有个**特别关键的手法**：配 `TMOD` 的时候不是直接 `TMOD = 0x01`，而是先 `&= 0xF0` 再 `|= 0x01`。为什么？因为 `TMOD` 这一个寄存器同时管着定时器0 和定时器1 的配置（高 4 位是定时器1，低 4 位是定时器0）。你要是直接 `TMOD = 0x01`，就把定时器1 的配置全冲掉了。`&= 0xF0` 先把低 4 位清零、保留高 4 位，再 `|= 0x01` 只设低 4 位——这是嵌入式里配寄存器的铁律：**只动你该动的那几位，别碰别人的位**。这背后就是第二章讲的位运算，在嵌入式里它是吃饭的家伙。

## 核心概念三：中断——硬件打断你的代码

超级循环有个毛病：`while(1)` 里如果在处理一个耗时的活，就顾不上别的了。可现实里硬件事件（定时器到点、按键按下、串口来数据）随时发生，不能等。这就得靠**中断**——硬件事件一来，CPU 自动暂停当前代码、跳去执行你写的中断服务函数，执行完再回来接着干。

8051 的中断服务函数长这样，Keil C51 用 `interrupt n` 这个特殊语法标明它是哪个中断向量的处理函数：

```c
void timer0_Routine() interrupt 1   // 1 = 定时器0 的中断号
{
    static unsigned int T0Count;
    TL0 = 0x18;          // 重装初值（模式1 每次溢出后要手动重装）
    TH0 = 0xFC;
    T0Count++;
    if (T0Count >= 1000) {   // 累计 1000 次 = 1 秒
        T0Count = 0;
        time.second++;       // 秒数进位
        ...
    }
}
```

注意三件事：第一，函数得在 `Timer0_Init` 里通过 `ET0 = 1; EA = 1;` 把中断打开，不打开硬件就算触发了也不会调你的函数；第二，定时器模式1 每次溢出后初值会清零，所以中断里第一件事就是**重装初值**（`TL0/TH0`），否则下一次定时就不准了；第三，`T0Count` 是 `static`，保证每次进中断它的值还在——普通局部变量每次进函数都重新分配，那计数就永远从 0 开始。

## 那个 1 毫秒是怎么算出来的

初值 `TH0=0xFC, TL0=0x18`（合起来 `0xFC18` = 64536）不是瞎写的，是算出来的。8051 的机器周期 = 时钟周期 ÷ 12，板子晶振 11.0592MHz，所以机器周期频率是 11.0592÷12 ≈ 0.9216MHz，一个机器周期约 1.085 微秒。定时器从初值往上数，数到 65536 溢出，所以一次溢出耗时 `(65536 - 64536) × 1.085μs ≈ 1.085ms`。代码里注释写的是 `"1ms@11.0592MHz"`，工程上这个近似够用了。累加 1000 次溢出，就是大约 1 秒——这就是这个时钟的计时原理。

## 真正的坑：这个 time 少了 volatile

讲到这里，必须给你点出一个**藏在这份真实代码里的陷阱**。`time` 这个结构体，在主循环里被读取（刷新显示），又在定时器中断里被修改（秒数进位）：

```c
TimeBuf time = {0, 0, 0, 0};   // ← 注意：没有 volatile！

// 主循环里读
showTime(time.second, time.minute, time.hours);
// 中断里写
void timer0_Routine() interrupt 1 { ... time.second++; ... }
```

问题在于，编译器不知道中断这回事，它看到主循环里反复读 `time.second`，又没看到主循环里有谁改它，就可能**自作主张把 `time.second` 缓存到寄存器里**，后面都用缓存值，再也不去内存里读——结果就是你眼睁睁看着秒数在中断里明明加了，LCD 上却死活不跳字。这种 bug 极其难查，因为代码逻辑一点没错，是编译器优化"帮了倒忙"。

解法就一个字：**`volatile`**。

```c
volatile TimeBuf time = {0, 0, 0, 0};   // 告诉编译器：这个变量随时会变，别缓存，每次老老实实去内存读
```

`volatile` 就是专门给这种"会被中断或硬件改"的变量准备的，它强制编译器每次都从内存重新读取，不做优化缓存。**在嵌入式里，只要一个变量同时被中断和主程序访问，给它加 `volatile` 是肌肉记忆**。这份代码没加，是个真实的缺陷，正好拿来当反面教材。

## 常见坑（真正的坑在后面）

> **坑 1：跨中断共享变量忘了 `volatile`。** 上面详述了。只要变量在中断里写、主循环里读（或反过来），一律 `volatile`。

> **坑 2：寄存器直接赋值覆盖了别的位。** `TMOD = 0x01` 会冲掉定时器1 的配置。永远用 `&=` 清位、`|=` 置位的方式只改目标位。

> **坑 3：中断里干重活。** 中断服务函数要尽快执行完，因为它在打断主程序。要是在中断里调一堆耗时函数、甚至用 `delay`，会堵塞其他中断、让主循环卡顿。正确做法是中断里只设个标志位、改个计数，真正的活留给主循环干。

> **坑 4：定时器模式忘了重装初值。** 模式1 是每次溢出后初值清零的，中断里不重装，下次定时就不准了。模式2（自动重装）才不用手动重，但精度和位数不同，按需选。

## 小结

- [ ] 裸机程序是超级循环：`void main()` + `while(1)`，main 不返回
- [ ] 操作硬件 = 读写寄存器（内存映射的固定地址）
- [ ] 配寄存器用 `&=` / `|=` 只改目标位，别整体赋值
- [ ] 中断用 `interrupt n`，进函数前要 `EA=1; ET0=1;` 开中断
- [ ] **跨中断共享变量必须 `volatile`**，否则编译器优化会让你读不到更新
